# AI 流程图项目 — 开发计划

## 一、产品定位

AI 驱动的代码 → PUML → 流程图在线编辑工具。

核心功能：
- 支持 C / C++ / Python 代码输入
- AI 自动将代码转换为 PlantUML 流程图
- 可视化流程图编辑（ReactFlow）
- 编辑后可反向生成 PUML 代码
- 导出 PNG / PDF / SVG
- 保存/加载历史项目

## 二、系统架构

```
┌─────────────────────────────────┐
│           Frontend              │
│  Vite + React 19 + TypeScript   │
│  ReactFlow + Monaco Editor      │
└────────────┬────────────────────┘
             │ REST / SSE
┌────────────▼────────────────────┐
│           Backend               │
│     FastAPI (Python 3.12)       │
└──────┬──────────┬───────────────┘
       │          │
┌──────▼──────┐ ┌─▼───────────────┐
│  AI Proxy   │ │ PUML Parser     │
│  (httpx)    │ │ (Python 实现)   │
└──────┬──────┘ └─┬───────────────┘
       │          │
       └────┬─────┘
            ▼
   ┌────────────────┐
   │  PostgreSQL     │
   │  + Redis Cache  │
   └────────────────┘
```

数据流：
1. 用户输入代码 → AI 转换为 PUML → 后端内部调用 puml_parser 解析为 Graph JSON → 一次性返回 puml + graph → ReactFlow 渲染
2. 用户手写/粘贴 PUML → 调用 POST /api/parse/puml → 返回 Graph JSON → ReactFlow 渲染
3. 用户编辑流程图 → Graph JSON → 前端 graphToPuml.ts 本地生成 PUML 代码（实时，无需请求后端）
4. 用户可随时保存/加载/导出

职责划分：
- PUML → Graph JSON（解析）：仅后端，前端不实现解析器
- Graph JSON → PUML（反向生成）：仅前端，编辑器内实时生成，无需后端参与
- AI 代码 → PUML：后端 AI Proxy 返回时，内部自动调用解析器，同时返回 puml 和 graph

## 五、核心数据模型

### Graph JSON（前端 + 后端共享）

```typescript
type NodeType =
  | "start"
  | "end"
  | "process"
  | "decision"
  | "loop"
  | "function_call"
  | "io"

type Node = {
  id: string
  type: NodeType
  label: string
  position: { x: number; y: number }
  data?: Record<string, unknown>
}

type Edge = {
  id: string
  source: string
  target: string
  label?: string
  type?: "yes" | "no" | "default"
}

type FlowGraph = {
  nodes: Node[]
  edges: Edge[]
  metadata?: {
    title?: string
    sourceLanguage?: "python" | "c" | "cpp"
    createdAt?: string
  }
}
```

### 数据库表（PostgreSQL）

```python
# users 表
class User:
    id: UUID (PK)
    email: VARCHAR (UNIQUE, INDEX)
    hashed_password: VARCHAR
    created_at: TIMESTAMP

# flow_projects 表
class FlowProject:
    id: UUID (PK)
    user_id: UUID (FK → users.id)
    title: VARCHAR
    graph_json: JSONB          # ReactFlow 数据
    puml_source: TEXT           # PUML 源码
    source_code: TEXT           # 用户原始代码
    source_language: VARCHAR    # "python" | "c" | "cpp"
    created_at: TIMESTAMP
    updated_at: TIMESTAMP
```

## 六、API 契约

### 1. AI 代码转换（SSE 流式）

注意：使用 POST + fetch ReadableStream 消费 SSE 流式响应，
Authorization Header 正常传递，token 不暴露在 URL 中。

```
POST /api/ai/convert
Authorization: Bearer <jwt>
{
  "code": "def hello():\n    print('hi')",
  "language": "python",
  "provider": "deepseek",
  "apiKey": "sk-..."
}

实现流程：
1. 后端根据 provider 选择对应 AIAdapter（openai / claude）
2. AIAdapter 通过 httpx 流式调用 AI API，逐块转发给前端（SSE）
3. 前端通过 fetch + ReadableStream 消费 SSE，支持自定义 Authorization Header
4. AI 返回完毕后，后端内部调用 puml_parser 解析完整 PUML
5. 最终 SSE 事件同时返回 puml 和 graph

Response (SSE 流式):
data: {"type": "chunk", "content": "@startuml\nstart\n"}
data: {"type": "chunk", "content": ":hello();\n"}
data: {"type": "done", "puml": "@startuml\nstart\n:hello();\nstop\n@enduml", "graph": {"nodes": [...], "edges": [...]}}

Response (非流式回退) 200:
{
  "puml": "@startuml\nstart\n:hello();\nstop\n@enduml",
  "graph": {
    "nodes": [...],
    "edges": [...]
  }
}

Response 422: AI 输出非法 PUML（重试 3 次后仍失败）
{
  "error": "PUML_GENERATION_FAILED",
  "message": "AI 输出无法解析为合法 PUML",
  "detail": { "raw_puml": "...", "parse_errors": [...] }
}

Response 502: AI 服务不可用
{
  "error": "AI_SERVICE_UNAVAILABLE",
  "message": "无法连接到 deepseek API",
  "detail": { "status_code": 503 }
}
```

```
Authorization: Bearer <jwt>

Request:

Response 200:
{
```


### 统一错误响应格式

所有接口的错误响应遵循以下格式：

```json
{
  "error": "ERROR_CODE",
  "message": "人类可读的描述",
  "detail": {}
}
```

常用错误码：
| HTTP 状态码 | error code | 含义 |
|-------------|-----------|------|
| 400 | BAD_REQUEST | 请求参数错误 |
| 401 | UNAUTHORIZED | 未认证或 token 过期 |
| 403 | FORBIDDEN | 无权限操作该资源 |
| 404 | NOT_FOUND | 资源不存在 |
| 409 | EMAIL_EXISTS | 邮箱已注册 |
| 422 | PUML_SYNTAX_ERROR | PUML 语法错误 |
| 422 | PUML_GENERATION_FAILED | AI 输出无法解析 |
| 429 | RATE_LIMITED | AI 调用频率超限 |
| 502 | AI_SERVICE_UNAVAILABLE | AI 供应商不可用 |
| 500 | INTERNAL_ERROR | 服务器内部错误 |

## 七、AI Prompt 设计

### 系统 Prompt

```
你是一个代码分析器，负责将源代码转换为 PlantUML 流程图。

规则：
1. 严格使用 PlantUML activity diagram 语法
2. 必须以 @startuml 开头，@enduml 结尾
3. 每个函数/方法独立为一个流程段
4. if/else 必须用 endif 完整闭合
5. 循环使用 while/endwhile 或 for/endfor
6. 函数调用使用 :call_function(); 格式
7. 输入输出使用 :read_input(); 或 :print_output(); 格式
8. 不要添加注释或解释，只输出 PUML 代码
```

### 用户 Prompt

```
将以下 {language} 代码转换为 PlantUML 流程图：

{code}
```

## 九、已知风险与应对

### 1. AI 输出不稳定

AI 生成的 PUML 可能语法错误。

应对：
- 后端对 AI 输出做语法校验，失败自动重试（最多 3 次）
- 前端解析失败时展示原始 PUML，允许用户手动修正
- Prompt 中严格约束输出格式

### 2. PUML 解析复杂

PlantUML 语法灵活，完整解析难度高。

应对：
- MVP 阶段只支持 activity diagram 基础语法
- 使用正则 + 状态机实现轻量解析器，不引入重型依赖
- 逐步扩展支持的语法范围

### 3. 画布性能

大量节点时 ReactFlow 可能卡顿。

应对：
- 使用 ReactFlow 的 virtualization 模式
- 超过 200 节点时自动折叠子流程
- 节点渲染使用 memo 包裹，避免不必要重绘

### 4. API Key 安全

用户自带 API Key，后端代理请求。

应对：
- Key 仅在请求期间存在于内存，不落库
- 强制 HTTPS 传输
- 后端不记录任何 AI 请求的 Key 内容

### 5. Redis 缓存策略

用途一：PUML 解析结果缓存
- key: `parse:{sha256(puml_text)}`，value: Graph JSON，TTL: 1 小时
- 相同 PUML 文本不重复解析，减轻 CPU 开销

用途二：AI 调用频率限制
- key: `ratelimit:{user_id}`，使用 Redis INCR + EXPIRE
- 限制：普通用户 10 次/分钟，避免滥用

### 6. 多 AI 供应商适配

不同供应商 API 协议差异大，需要适配层隔离：

| 供应商 | 端点 | 认证方式 | 流式格式 |
|--------|------|----------|----------|
| OpenAI | /v1/chat/completions | Authorization: Bearer | SSE, data: [DONE] |
| DeepSeek | /v1/chat/completions | Authorization: Bearer | SSE, data: [DONE]（兼容 OpenAI） |
| Claude | /v1/messages | x-api-key + anthropic-version | SSE, event: message_stop |

设计：
- 定义 AIAdapter 抽象接口（chat_stream 方法）
- OpenAIAdapter 处理 OpenAI + DeepSeek（协议相同）
- ClaudeAdapter 处理 Claude（协议不同）
- ai_proxy.py 根据 provider 字段选择适配器
